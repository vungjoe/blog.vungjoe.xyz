Django中的Field types

发表于 2019-08-13 | 分类于 WEB 开发

Field types

下面是 Django 中的常见字段类型及其说明，

一、Django 内置字段类型（不包括关系字段类型）

1.1 AutoField

AutoField 是一个自动增加的整数类型字段。通常不需要自己编写它， Django 会自动帮你添加字段 id = models.AutoField(primary_key=true) 。

  • 这是一个自增字段，从 1 开始计数
  • 如果需要自行创建主键，务必将该字段设为 primary_key=true
  • Django 在模型中只允许有一个自增字段，且该字段必须为主键

1.2 BooleanField

BooleanField 布尔值类型，默认是 Nono 。如果需要 Null 为某字段的值，请使用 NullBooleanField 。

1.3 CharField

CharField 字符串类型，必须接受一个 max_length 参数。表示字符串长度不能超过该值。最最常用的类型！

1.4 DateField

DateField 日期类型，

1.5 DecimalField

固定精度的十进制小数。相当于Python的Decimal实例，必须提供两个指定的参数！

  • 参数max_digits：最大的位数，必须大于或等于小数点位数
  • decimal_places：小数点位数，精度

1.7 FilePathField

FilePathField ，一种用来保存文件路径信息的字段。

1.8 GenericIPAddressField

IP 地址，包括 IPv4 和 IPv6 ，字符串形式

1.9 ImageField

如需使用 ImageField ，需提前安装 pillow 模块。使用 pip install pillow 即可。

ImageField 图像类型，用于保存图像文件的字段。其基本用法和 FileField 相同，只不过多了两个属性 height 和 weight 。

  • 在数据库中，实际保存的是一个字符串类型，默认最大长度 100
  • 真实的图像文件保存在服务器的文件系统内

安全建议：

  • 无论如何保存上传的文件，一定要注意他们的内容和格式，避免安全漏洞！
  • 务必对所有上传的文件进行安全检查，确保他们不出安全问题。

1.10 IntegerField

整数类型，最常用的字段之一。取值范围 -2147483648 到 2147483647 （ 2 的 32 次方分开两半）。

1.11 TextField

大量文本内容，在 HTML 中表现为 Textarea 标签，最常用的字段类型之一！如果你为它设置一个 max_length 参数，那么在前端页面中会受到输入字符数量限制，然而在模型和数据库层面却不受影响。只有 CharField 才能同时作用于两者。

1.12 URLField

一个用于保存URL地址的字符串类型，默认最大长度200。

1.13 UUIDField

用于保存通用唯一识别码（Universally Unique Identifier）的字段。

  • 使用 Python 的 UUID 类。在 PostgreSQL 数据库中保存为 uuid 类型，其它数据库中为 char(32)。
  • 这个字段是自增主键的最佳替代品。

二、字段的参数

所有的模型字段都可以接收一定数量的参数，比如 CharField 至少需要一个 max_length 参数。下面的这些参数是所有字段都可以使用的，并且是可选的。

2.1 null

  • 默认为 False
  • null 为 True 时，Django 在数据库中用 NULL 保存空值。
  • 对于字符串类型，尽量避免将此选项设为 True ，因为这样会导致两种没有数据的情况，一种是 NULL ，一种是“空字符串”

2.2 blank

  • 默认为 False
  • 当设为 True 时，表示字段可以为空
    • 和 null 参数不同， null 是纯数据库层面的，而 blank 是验证相关的，它与表单验证是否允许输入框内为空有关，与数据库无关。
  • 当 null 为 True 、blank 为 false 时，如果字段接收到空值可能会出现异常。

2.4 db_column

  • 该参数用于定义当前字段在数据表中的列名，
  • 如果未指定，Django 将使用字段名作为列名

2.5 db_index 和 db_tablespace

db_index

  • 该参数值是布尔值
  • 如果该参数值为 True ，数据库将为该字段创建索引

db_tablespace

  • 用于字段索引的数据库表空间的名字，前提是当前字段设置了索引。
  • 默认值为工程的 DEFAULT_INDEX_TABLESPACE 设置。如果使用的数据库不支持表空间，该参数会被忽略。

2.6 default

  • 字段的默认值，可以是值或者一个可调用对象。
    • 如果是可调用对象，那么每次创建新对象时都会调用。设置的默认值不能是一个可变对象，比如列表、集合等等。
    • lambda 匿名函数也不可用于 default 的调用对象，因为匿名函数不能被 migrations 序列化。

2.7 editable

  • 参数默认值为 True
  • 如果设为False，那么当前字段将不会在admin后台或者其它的ModelForm表单中显示，同时还会被模型验证功能跳过。

2.9 error_message

  • 用于自定义错误信息
  • 该参数接收字典类型的值，字典的键可以是 null 、 blank 、 invalid 、 invalid_choice 、 unique 和 unique_for_data 中的一个

2.10 help_text

  • 额外显示在表单部件上的帮助文本
  • 使用时请注意转义为纯文本，防止脚本攻击

2.11 primary_key

  • 如果没有给模型的任何字段设置此参数为 True ， Django 将自动创建一个 AutoField 自增字段，名为 “id”，并设置为主键也就是 id=models.AutoField(Primary_key=True)
  • 如果你设置了某个字段的 Primary_key=True ，则当前字段变为主键，并关闭 Django 自动生成 id 主键的功能
  • 主键字段不可修改，如果你给某个对象的主键赋个新值实际上是创建一个新对象，并不会修改原来的对象。

2.12 unique

  • 当 unique=True 时，在整个数据表内该字段的值不能重复
  • 对于 ManyToManyField 和 OneToOneField 两个关系类型，该参数无效
  • 当 unique=True 时， db_index 参数无需设置，因为 unique 隐含了索引

2.13 unique_for_data 、 unique_for_month 和 unique_for_year

  • 日期唯一
  • 可能不太好理解。举个例子，如果你有一个名叫 title 的字段，并设置了参数 unique_for_date="pub_date" ，那么 Django 将不允许有两个模型对象具备同样的 title 和 pub_date
  • 有点类似联合约束。
  • unique_for_month 和 unique_for_year 的作用和 unique_for_data 类似，只不过是月份唯一和年唯一

2.14 verbose_name

  • 为字段设置一个人类可读，更加直观的别名。
  • 对于每一个字段类型，除了 ForeignKey 、 ManyToManyField 和 OneToOneField 这三个特殊的关系类型，其第一可选位置参数都是verbose_name
    • 对于外键、多对多和一对一字字段，由于第一个参数需要用来指定关联的模型，因此必须用关键字参数verbose_name来明确指定
  • 如果没指定这个参数，Django会利用字段的属性名自动创建它，并将下划线转换为空格。

2.15 validators

  • 运行在该字段上的验证器的列表。

三、关系字段类型

使用关系字段的基本原则

  1. 一对一的表，两表的属性实际上完全可以合并为一个表，共用一个主键即可；
  2. 一对多的表，可以设中间关联表，也可以将关联表并入“多”这头；若设独立关联表，则可引入“多”这头的主键作为其主键，也可另立主键并将“一”和“多”两表的主键做为关联表的外键；
  3. 多对多的表，则必须设中间关联表（Django 会自动生成，也可以自定义），关联表设独立主键，并引入两个“多”头的表的主键作为关联表的外键；
  4. 能用一对一的，就不用一对多；能用一对多的，就不用多对多，尽量简化；
  5. 能当作属性处理的，尽量当作属性，而不是当实体处理去另立新表，这样可使问题简化；
  6. 把意义相近、联系紧密的属性放在一张表内，而不是拆分在多个表中。

3.1 多对一/外键（ ForeignKey ）

多对一关系，通常被称为外键。外键字段类的定义如下：

参数说明：

  • 外键要定义在【多】的一方
  • 外键需要两个最重要的位置参数，一个是关联的模型，另一个是 on_delete 选项
    • to 就是表示该字段关联到哪个模块，用单引用号括起来
      • 可以引用其他应用的模块，比如 ForeignKey(‘appname.modelname’,on_delete=…)
    • on_delete ，表示当一个被外键关联的对象被删除时，Django 将执行 on_delete 参数定义的 SQL 约束执行相应的操作。Django 2.2 一共有 6 个可选项
      • CASCADE：模拟 SQL 语言中的 ON DELETE CASCADE 约束，将定义有外键的模型对象同时删除（此为 Django 2.2 的默认操作）
      • PROTECT：阻止上面的删除操作，弹出 ProtectedError 异常
      • SET_NULL：将外键字段设为 null。只有当字段启用了 null=True 时才可以使用本选项
      • SET_DEFAULT：将外键段设为默认值。只有当字段设置了 default 参数时，方可使用
      • DO_NOTHING：什么也不做
      • SET()：设置为一个传递给 SET() 的值或者一个回调函数的返回值。注意大小写。

上面介绍的最基本的两个参数，其他参数的说明如下：

  • limit_choices_to：该参数用于限制外键所能关联的对象，只能用于 Django 的 ModelForm （表单模块）和 admin 后台，对其他场合无限制功能。
  • related_name：用于关联对象反向引用模型的名称。
    • 通常情况下，这个参数无需我们进行设置，Django 会默认以模型的小写加上 _set 作为反向关联名。
    • 这个参数就是在使用“一”的时候，需要引用“多”头的数据，这时就使用 一.多_set，就可以关联到“多”头了
  • related_query_name：反向关联查询名。用于从目标模型反向过滤模型对象的名称。
  • to_field：默认情况下，外键都时关联到被关联对象的主键上（一般为 id）。如果指定该参数，则可以关联到指定的字段上，但该字段必须具有 unique=True 属性，也就是具有唯一属性
  • db_constraint：默认值为 True，表示遵循数据库约束，绝大多数情况下，使用默认值就可以了。
    • 如果设为 False，那么将无法保证数据的完整性和合法性。该参数设为 False 时，当访问一个不存在的关系对象时，会抛出 DoesNotExist 异常。
    • False 的使用场景：
      • 有历史遗留的不合法数据，没办法的选择
      • 你正在分割数据表
  • swappable：控制迁移框架的动作，如果当前外键指向一个可交换的模型。使用场景非常稀少，通常只需保持默认 True 即可。

3.2 多对多（ManyToManyField）

多对多关系在数据库中也是非常常见的关系类型。比如一本书可以有好几个作者，一个作者也可以写好几本书。

  • 多对多的字段可以定义在任何的一方，请尽量定义在符合人们思维习惯的一方
  • 不要同时都定义

多对多关系类的定义如下：

与外键类似，多对多关系也需要一个最重要的位置参数：即关联的对象模型。其用法与外键的用法基本一致。

在数据库后台，Django 实际上会额外创建一张用于体现多对多关系的中间表。默认情况下，该表达名字是“多对多字段名+关联对象模型名+一个独一无二的哈希码”，比如“author_books_fh02f”。当然，也你也可以通过 db_table 选项，自定义表名。

3.2.1 参数说明

  • related_name：参考外键的相同参数
  • related_query_name：参考外键的相同参数
  • limit_choices_to：参考外键的相同参数。但是对于使用 through 参数自定义中间表的多对多字段无效
  • sysmmetrical：默认情况下，多对多关系是对称的
  • through：定义中间表，该参数的值为一个中间模型。
  • through_fields：如果中间表包含了两个关联同一模型的外键，则必须使用该参数显式地指定关联对象。用于定义关系。这个参数放在下面的例子中帮助理解。
  • db_table：设置中间表的名称，不指定时，使用默认值“多对多字段名+关联对象模型名+独一无二的哈希码”。
  • db_constraint:参考外键的相同参数.
  • swappable：参考外键的相同参数。
  • 多对多关系字段不支持 Django 内置的 validators 验证功能。
  • null 参数对多对多字段无效，设置 null=True 毫无意义。
